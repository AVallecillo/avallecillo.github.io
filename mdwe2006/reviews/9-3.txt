- Summary and main contributions:

=> Motivation:

The WebML modeling language, which is a Domain Specific Language for modeling Web applications, is not based on current modeling standards, i.e., MOF and UML. Yet, there is an increasing interest for interoperability with other notations and tools.

=> Solution:

This paper proposes a UML 2.0 profile for WebML.

=> Methodology:

The authors first concentrate on WebML's concepts by defining a MOF-based metamodel for WebML. In a second step, they identify appropriate UML meta-classes in order to derive stereotypes for each WebML concept.

The profile is then explained using the CD store example by comparing WebMLs's original data model and hypertext model representations with the corresponding UML profile representations.







- Weak points of the paper:

A related work section is missing! There have already been proposals for web modeling metamodels (e.g., for W2000 and UWE)



No information on how the metamodel has been derived, in particular, if all WebML concepts have been incorporated in the metamodel, if additional concepts have been introduced.



It is not explained which elements from figure 5 belong to which packages from figure 4.







- Strong points of the paper:

CD store modeling example to illustrate the UML profile, comparison with original WebML models







- Any suggestions for improvements:

***Questions***

You provide no information if there is a one-to-one mapping between the original WebML concepts and the concepts currently present in the WebML metamodel, e.g., why has the PageElement (Figure 5) meta-class been introduced?

=> How did you derive the metamodel from the existing WebML concepts. What was your concrete methodology?



Section 1

Doesn't the OMG provide 3 mechanisms for defining DSLs?

1) a new MOF-based metamodel (example W2000 metamodel, maybe a more appropriate example than the CWM)

2) heavy-weight extension of the UML metamodel (example: UWE metamodel)

3) light-weight extension with UML profiles (example: Profile of Conallen)



Section 2.1

"Regarding WebML data types, we have metamodeled them using MOF metaclasses for constructing primitive types. Consequently, with the exception of primitive types like Integer and String, all WebML types have been defined as user-defined types."

=> Unfortunately, I was not able to understand the meaning of this sentence. How does this sentence related to the enumeration WebMLType, to the class WebMLType and to the class Domain?



Figure 1

Could you please provide information which elements are grouped within the PresentationView package?



Figure 2

ModelElement should be italic, since it is an abstract meta-class (according to the accompanying text). Shouldn't some of the other meta-classes also be abstract and thus depicted italic (such as IdentifiedElement etc.)?



Figure 3

- Modeling that an entity has got an OID attribute: I would suggest modeling an extra 1:1 association between entity and attribute and a role name of "OID" and changing the multiplicity from the existing association from 1..* to 0..*.

- If you really want to introduce the new concept "RelationshipRole" with multiplicity 2, the "inverse" association for "Relationship" is not necessary anymore.

- Isn't there an attribute of type <<enumeration>> WebMLType missing in class WebMLType?





Section 3.3

Selecting content from the datamodel seems to work according to a certain pattern (<<focus>> class, ...). Could you explain more explicitely the purpose of, for example, the <<focus>> stereotype.



Figure 5

How do you cope with the different semantics of OKLinks & KOLinks and AutomaticLinks & TransportLinks?

You do not provide information on which elements of WebML have been included and which elements have currently been omitted in the metamodel, e.g., the "alternative page" concept is missing. Although mentioned in the text, the "selector" concept is not represented in the metamodel either.

Figure 6

I'm aware of your motivation for using classes instead of associations for represention relationships in the UML profile. Still, I think that you have to state your reasons more explicitly, maybe present an example.

 - Why don't you consider using association classes instead of normal classes. I'm sure you are aware of the semantic difference between using association classes and normal classes. But wouldn't association classes be more appropriate for specifying the unique character of your relationships? If not, please explain it to the reader.
