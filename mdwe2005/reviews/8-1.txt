This paper is very interesting and certainly worth to be presented in the
workshop. With improvements, it will be an excellent paper that might be
published elsewhere. But it is not easy to read, due to two reasons: the
English should be improved, and there are many small inconsistencies (e.g. in
fig.5, fig. 6 and the text of section 5.1).


The described approach is very ambitious, and the presented transformations, in
particular T1, are very general. It seems that they will be very complex, in
detail. Thus, I have doubts whether they can be realized as described, without
the system becomming overly complex. Therefore, the paper should state clearly
if it describes ideas and concepts, or if running (prototype) implementations
exist already.


Here my questions and remarks:


Is it intended that WebSA generates the complete code of a Web application, or
only class skeletons? If it is to generate the complete code, it is not clear
how this is done, since the semantics of the operations of the conceptual and
navigational model are not defined.


The configuration model is defined by a composite structure model from UML 2.0.
An important characteristic of the composite structure model is that components
have provided and required ports with interfaces that define sets of
operations.
In fig.4, the sets of operations are not defined, only the names of the
interfaces are given. It is not clear which operations each of these
interfaces, like e.g. a general model data interface, defines. In fig.7, the
interfaces have names different from those in fig.4. Where do these interfaces
come from? Are they generated by T1? If so, by which transformation rule (of
those presented in fig.5)? Another problem is that UML components with
different interfaces are different components. That means a trip model
component from fig.7 does not be an instance of a model component from fig.4.
Further, in fig.7, a model component has model data as a required interface,
whereas in fig.4, model data is a provided interface of the model component.


The paper states that the architecture of the generated Web application is
defined by the subsystem model and the configuration model and not by the
transformation T1. This would be a very powerful and general approach. But from
my point of view, the approach is very complex, and I have doubts if it works
without making very restrictive assumptions. The point is that the
transformation T1 must be defined independently from the subsystem model and
the configuration model so that the approach works: a change of the
configuration model should not necessitate a change of T1. However, this does
not seem to be the case. For example, the transfomation described in Fig.6
refers to server pages; if the configuration model does not contain server
pages, but e.g. servlets, the transformation T1 has to be modified. So my
impressin is that different T1 transformations are required for essentially
different configuration models.


The paper states that the integration model is a PIM. I disagree; from my point
of view, it is a PSM. The integration model contains server pages and MVC2, as
described in Fig. 4 and 7; but those form a special technology platform, so
that the integration model is a PSM. The integration model described in Fig. 7
may be implemented on several vendors platforms, like JSP or ASP, but it could
certainly not be transformed meaningfully to CORBA which is included as a
possible target platform in section 7. This shows that fig.7 does not
constitute a PIM. Further, T2 should be considered as a PSM to code
transformation.


The integration model of Fig. 7 shows a module Logic Presentation. It is not
clear where this module originates from, since there is no such component in
the configuration model nor in the conceptual model?


Overall, the approach is very interesting and promising, but the paper leaves
many questions open!
