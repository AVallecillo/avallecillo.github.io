- Summary and main contributions:
  This paper motivates the need for modelling the navigation in web
  applications and introduces a pipe-based definition of concepts to tackle
  this problem.
  The navigation model is tightly coupled with other potentially involved
  models for web engineering. In particular, it is well addressed how the
  newly introduced model does not reduce the compatibility with existing UML
  models.
  The traditional example of the Travel Agency is fully explained and
  examplified, along with the UML-Conallen model.
  The concepts introduced in the paper is really fine-grained so that it is
  possible to define a navigation not only at a higher level of abstraction
  than the code level, but also at a finer grain than existing models.


- Weak points of the paper:
  The modeling elements defined in section 3.2 are appropriate and sound,
  but the mapping between these elements and the widgets used to support these
  concepts are defined in an immuable way. For instance,
  a nexus mode is mapped onto a window. This mapping is too much physical.
  Could we imagine to map it onto a family of widgets that correspond to the
  expected behavior such as a window, a dialog box, a pane, a frame. In this
  case, a container node
  could be not only a frame, but also a group box, a region delineated with
  a separator, etc. Similarly, a push button is not the only way to trigger a
  function.
  It could be nice to introduce a little bit some comparison between the
  Pipe notation and some other notation, like WebML which also pretty much
  addresses the same kind of problem. In this way, one could better identify
  where and how the Pipe notation
  does a better job than other models.


- Strong points of the paper:
  The model introduced for navigation clearly distinguishes the static part
  from the dynamic part. It also does a very good job in integrating the
  different models involved in the methodology and does cover the user
  interface of the web application in connexion with the set of the models.
  The paper is not limited to the navigational model itself, it also
  addresses how to combine it with other models, like the UML-Conallen one.
  In addition, implementation issues are also addressed thus making this
  paper very appropriate for a discussion of the results provided.
  The question of scalability is a traditional one: if the model becomes
  very large, is it still tractable and what are the consequences on the
  viability of the rest of the development life cycle?
