The paper reports hard results and an interesting conjecture.  It would be more valuable if more information about the techniques for code generation were provided.  The interesting conjecture is about why different models generated different amounts of code.  The relationship to ODP is not strong.

- Summary and main contributions:
The authors defined a small extension to UML in the style of OOSE, built tools for specifying applications using the extended UML and generated code from the models.  The contribution is to report on code sizes generated from differing models.

- Weak points of the paper:
Does not tell us as much as it might about the code generation technology used.

- Strong points of the paper:
Hard numbers about code generation.

An interesting conjecture about the amount of code generated by different parts of the model.

5) FURTHER COMMENTS FOR AUTHORS:

"An entity class is for information viewpoint. A control class is for computational viewpoint."  This suggests the misunderstanding of the ODP viewpoints sometimes encountered in the literature.  It is not the case that the information viewpoint is about data and the computational viewpoint is about behavior.  The dynamic schemata of the information viewpoint are behavior specifications.  The computational objects must encapsulate all of the data in the system.

Explain why there is a reference to SDL ([4]) or remove the reference.

6) AFTER DISCUSSION IN THE PROGRAM COMMITTEE:

After discussion, I'm willing to reject this paper.  I agree the connection to ODP is not there.  I hope the paper can be accepted elsewhere.  We need to see some work along these lines.
